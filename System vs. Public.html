External DTDs: System vs. Public

Introduction

An external DTD (Document Type Definition) is a separate file used to define the structure and rules for an XML document. External DTDs can be declared as either SYSTEM or PUBLIC, depending on how and where they are accessed. Hereâ€™s a small research summary on the differences between SYSTEM and PUBLIC external DTDs:

1. SYSTEM DTD

A SYSTEM identifier specifies the location of the DTD file directly using a URL or a file path.

  • Declaration Syntax:

    <!DOCTYPE root-element SYSTEM "file-or-URL">

  • Characteristics:

    • The identifier is a URI or file path that points to the exact location of the DTD.
    • It is straightforward but tightly couples the XML document to the location of the DTD.
    • Suitable for private or localized usage where the DTD is not meant to be shared broadly.

    • Example:

      <!DOCTYPE note SYSTEM "note.dtd">

      Here, the XML parser will look for note.dtd in the specified path.

  • Advantages:

    • Simple and easy to implement.
    • Works well for internal projects or specific environments.

  • Disadvantages:

    • The exact location of the DTD must be accessible; changes in the file's path or server can break validation.
    • Less portable for widely shared XML files.

2. PUBLIC DTD

A PUBLIC identifier uses a public identifier string to refer to the DTD, often in combination with a SYSTEM identifier as a fallback.

  • Declaration Syntax:

    <!DOCTYPE root-element PUBLIC "public-identifier" "file-or-URL">

  • Characteristics:

    • The public identifier is a formal identifier (e.g., a registered name or string) that describes the DTD.
    • It often includes a SYSTEM identifier as a backup, which specifies the location of the DTD.
    • Suitable for standardized, widely shared DTDs where many systems can recognize the public identifier.

    • Example:

      <!DOCTYPE html PUBLIC "-//W3C//DTD XHTML 1.0 Transitional//EN" "http://www.w3.org/TR/xhtml1/DTD/xhtml1-transitional.dtd">

  • Advantages:

    • Promotes portability and standardization.
    • Useful for DTDs that are intended for widespread use, such as HTML or industry-specific standards.
    • The XML parser can use local caches or resolvers for public identifiers to improve performance.

  • Disadvantages:

    • More complex to implement and maintain.
    • Requires coordination to ensure that public identifiers are standardized and resolvable.

3. Comparison Table

Aspect SYSTEM PUBLIC
Usage Scope Private, localized use Standardized, public use
Identifier Specific file path or URL Public identifier + optional fallback
Portability Limited High
Complexity Simple Moderate to complex
Examples note.dtd HTML DTDs, industry schemas


4. Conclusion

  • Use SYSTEM for local or project-specific XML files where the DTD location is fixed and controlled.
  • Use PUBLIC for standardized or widely used DTDs to ensure better portability and flexibility, especially when sharing across systems or organizations.
